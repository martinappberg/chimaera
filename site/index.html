An agent workbench,
not an IDE.

chimaera runs your coding agents as persistent, daemon-owned sessions on whatever host owns the work — your laptop, a dev server, or an HPC login node — and wraps them in a workspace-first UI. Close the lid mid-run and nothing dies.

The problem

Two tools, duct-taped, and still losing your work

The usual way to run agents on a remote box is a split stack: code-server in a browser tab to look at files, tmux over SSH to keep the agents alive — one nested inside the other. Browser terminals die on reload. tmux renders nothing. And the real deliverable of an agent run is files — reports, plots, tables — that neither tool puts next to the session that made them.

What you get

The whole stack, in one window that survives disconnection

Sessions that survive disconnect

tmux-grade ownership: the daemon holds every PTY with full server-side terminal state. Reload, reconnect, or reattach from another machine and get the identical screen back — no lost scrollback, no broken reconnect.

Multi-agent launcher

Claude Code, Codex, Gemini CLI and Antigravity in one launcher — detected if installed, resumable per workspace, installable and updatable from official sources. Binaries live under ~/.chimaera; credentials stay entirely yours.

Real interactive TUIs

Agents run as the actual claude TUI in a daemon-owned PTY — so they look, behave, and bill exactly like they do in your terminal. Hooks turn their state into attention signals so you know which one needs you.

File previews that replace code-server

Images, Markdown, CSV/TSV (gzip included), PDF, notebooks, and sandboxed self-contained HTML reports — the MultiQC/FastQC class of outputs. The server extracts, the client renders; whole files are never loaded.

A real workbench layout

Split panes, tabs, drag-and-drop, focus mode, and a session strip that always says where you are. Terminals and file previews are the same kind of surface — arrange them however the work wants.

Linked terminals

Hand an agent a leash to a live shell — the one with your environment already loaded — instead of paying setup cost per command. Links are user-granted, scoped, and audited in the terminal's own scrollback.

How it works

One binary owns the sessions. Windows are just views.

Local or remote

Connect over your own SSH

chimaera connect <host> shells out to your system ssh, so ProxyJump, ControlMaster, and 2FA from ~/.ssh/config all just work. The daemon installs itself into ~/.chimaera on the host — no root, no containers, nothing else to set up.

  • Install-if-missing. Detects the target with uname and drops in the right static binary.
  • Start-or-attach. Reuses a running daemon; tunnels the port; opens the UI.
  • Self-updating. An outdated remote daemon is replaced on connect — never killing a shell that holds a live environment without asking.
zsh — your laptop 
# start it locally
$ chimaera serve
  → workbench on http://127.0.0.1:9700

# or drive a remote host — HPC login node, dev box, anything
$ chimaera connect gpu-node
  → installing daemon in ~/.chimaera … ok
  → tunnelling :9700 … ok
  → opening workbench ✓
Built for real infrastructure

One static binary. Old glibc. No root.

The daemon cross-compiles to fully static musl binaries (x86_64 and aarch64) that run on ancient HPC systems — verified on a cluster running glibc 2.17, no root required. It's designed to live on shared login nodes: bounded allocations, hard ceilings on preview extraction, ~150 MB RSS steady-state.

  • Durable, NFS-safe state. Append-only, size-capped JSONL under ~/.chimaera — no SQLite anywhere near Lustre.
  • Good citizen. No busy loops, no unbounded buffers. Resource discipline is a review criterion.
  • Native app included. A Tauri shell wraps the same UI — a window per workspace, one-click connect, auto-updating. Quitting kills nothing.
login-node — CentOS 7.9 
$ ldd ./chimaera
  not a dynamic executable

$ ./chimaera serve --port 9700
  chimaera 0.1 · build a1b9f3c
  workspaces: 2 · sessions: 5 (3 live)
  listening on 127.0.0.1:9700

$ ssh gpu-node # close the laptop — nothing dies

Questions

Good to know

Does it change how my agents run or bill?
No. Agents run as their real interactive TUIs in a normal PTY, spawned through your login shell with your environment and credentials. claude behaves and bills exactly as it does in any terminal — chimaera just owns the process and gives you a window onto it.
Do my API keys or credentials go through chimaera?
No. Credentials stay entirely yours — they live in your HOME and login-shell environment, independent of who installed the agent binary. chimaera can install and update agent CLIs from official sources, but never touches your auth.
Is it only for HPC?
No — it's a general tool for running agents wherever the work lives. HPC login nodes are the founding stress test (old glibc, no root, shared machines), which is exactly why it also behaves well on your laptop or a plain dev server.
What platforms are supported?
macOS and Linux. The daemon ships as static musl binaries for x86_64 and aarch64 Linux plus an aarch64 macOS build; the native app is macOS (Apple Silicon). Windows is untested. See the docs for details.
What's the license?
GNU AGPL-3.0 — free to use, self-host, and modify. If you want to build a closed-source product or service on chimaera, a commercial license is available: get in touch. Contributions require a CLA.
Is it production-ready?
It's early and moving fast. Interfaces, storage formats, and the wire protocol can change without notice. It's genuinely useful today — but pin your version and read the DESIGN.md before you depend on it.
